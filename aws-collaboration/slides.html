Fire-and-Forget Architecture

Async execution model & agent topologies

Developers
Developers

Why Fire-and-Forget?

The Challenge

  • LLM inference is slow - 30-120 seconds per agent
  • Lambda timeout - 15 min max integration
  • Agents are stateless - No persistent connections
  • Need caching - Don't re-run unchanged inputs

The Solution

  • Task tokens - Step Functions waits up to 1 year
  • DynamoDB streams - Event-driven callbacks
  • Deterministic session IDs - Enable caching
  • Decoupled execution - Fire and forget
Key Insight: Step Functions pauses, Lambda returns immediately, agent runs async, DynamoDB stream triggers callback on completion.

What is a Session ID?

Definition

A deterministic identifier that correlates all artifacts for a single manuscript evaluation.

backend/functions/start_workflow/lambda_function.py 
def generate_session_id(manuscript_id, model_id):
    return hashlib.sha256(
        f"{manuscript_id}:{model_id}".encode()
    ).hexdigest()

# Example:
# manuscript_id = "BMJ-2024-12345"
# model_id = "claude-sonnet-4-5..."
# → session_id = "a1b2c3d4..."

Key Properties

  • Deterministic: Same inputs → same session_id
  • Content-addressable: Represents input combination
  • Collision-resistant: SHA-256 ensures uniqueness

Enables

  • Caching Skip re-execution
  • Deduplication Prevent duplicates
  • Correlation Find related data
  • Replay Re-run without loss

AGENT INVOCATION (agentcore_stepfunctions_invoke Lambda) ┌────────────────────────────────────────────────┐ │ DynamoDB: agent-sessions │ │ PK: session_id="a1b2c3..." │ SK: agent_name │ │ task_token, status, s3_result_key │ └────────────────────────────────────────────────┘ ┌────────────────────────────────────────────────┐ │ S3: scoping/a1b2c3.../ │ │ ├── payload.json (input) │ │ └── result.json (output) │ └────────────────────────────────────────────────┘ 4. AGENTCORE: runtimeSessionId = session_id 5. COMPLETION: write_session_complete(session_id, result) 6. RETRIEVAL: GET /agents/{session_id}/results

Constraining LLM Outputs with Pydantic

Define Output Schema

backend/agents/scoping/models.py 
from pydantic import BaseModel, Field
from typing import Literal

class ScopingDecision(BaseModel):
    """Scoping decision output schema."""

    topic_fits_journal: Literal["Yes", "No", "Uncertain"] = Field(
        description="Whether topic fits journal scope"
    )
    study_type_fits_journal: Literal["Yes", "No", "Uncertain"] = Field(
        description="Whether study type is accepted"
    )
    decision: str = Field(
        description="Keep/Transfer/Reject/Further Investigation"
    )
    reasoning: str = Field(
        description="Explanation of assessment"
    )

Key Pydantic Features

Literal[...] Constrain to specific values
Field(description=) Guide LLM with semantics
list[str] Require list of strings
Optional[...] Allow null values
Nested models Compose complex structures
Result: result.structured_output is a typed ScopingDecision instance

Creating an Agent with Constraints

backend/agents/scoping/workflow_agent.py 
from strands import Agent
from strands.session import FileSessionManager
from strands_tools import file_read
from models import ScopingDecision
from utils import create_bedrock_model

class WorkflowScopeAgent:
    def initialize(self, session_id: str = None):
        # 1. Create Bedrock model connection
        model = create_bedrock_model(
            model_id="eu.anthropic.claude-sonnet-4-5-20250929-v1:0",
            temperature=0.1,          # Low for consistency
            cache_prompt="default",   # Enable prompt caching
        )

        # 2. Load system prompt (instructions for LLM)
        decision_prompt = self._load_prompt("decision_prompt.txt")

        # 3. Create agent with structured output constraint
        self._decision_agent = Agent(
            model=model,                              # Bedrock connection
            system_prompt=decision_prompt,            # LLM instructions
            tools=[file_read],                        # Available tools
            structured_output_model=ScopingDecision,  # ← OUTPUT CONSTRAINT
            session_manager=FileSessionManager(session_id),
            hooks=self.hooks,                         # Event callbacks
        )

You are a Scoping Specialist for {journal_name}.

Your task: Evaluate if a manuscript fits {journal_name}
based on BOTH topic relevance AND study type.

## Available Tools

Use `file_read` to access:
- journal_scopes/{journal}/scope_summary.txt
- journal_scopes/{journal}/article_types.txt

### 1. Topic Fit (topic_fits_journal)
- **Yes**: Primary focus aligns with scope
- **No**: Belongs to different specialty
- **Uncertain**: Mixed or unclear focus

**Keep for {journal_name}**: Both topic AND study type fit
**Transfer**: Better fit for another BMJ journal
**Reject**: Not related to health outcomes

Prompt Best Practices

  • Template variables: {journal} replaced at runtime
  • Document tools: Tell LLM what tools are available
  • Define criteria: Match Pydantic field names
  • Decision rules: Clear logic for each outcome
  • Few-shot examples: Append JSON examples for guidance
Key: Prompt field names should match Pydantic model fields exactly

Complex Example: Nested Pydantic Models

backend/agents/ethics/models.py 
class CriterionEvidence(BaseModel):
    """Evidence for a specific ethics criterion."""
    criterion: str = Field(description="Requirement from guideline")
    source_url: str = Field(description="URL where found")
    mandatory: bool = Field(description="Whether mandatory")
    evidence_found: Optional[str] = Field(
        description="Quote from manuscript, or null"
    )
    compliance: Literal["low", "medium", "high"]

class EthicsAssessment(BaseModel):
    """Ethics compliance assessment output."""
    compliance_level: Literal["low", "medium", "high"]
    decision: Literal["accept", "revise", "reject"]

    criteria_evaluated: List[CriterionEvidence] = Field(
        default_factory=list
    )  # ← Nested model list

    ethics_approval_mentioned: bool
    consent_procedures_adequate: bool
    issues_found: List[str] = Field(default_factory=list)
    recommendations: List[str] = Field(default_factory=list)
    reasoning: str

Pattern Benefits

  • Type safety: All fields validated
  • Nested structures: List[CriterionEvidence]
  • Boolean flags: Clear yes/no checks
  • Enum constraints: Literal[...] for valid values
  • Defaults: default_factory=list for empty lists

Result Usage

ethics = result.structured_output
if ethics.compliance_level == "low":
    reject()
for issue in ethics.issues_found:
    log(issue)

Loading & Templating Prompts

Load from File

backend/agents/scoping/workflow_agent.py 
def _load_prompt(self, filename: str) -> str:
    """Load prompt from file."""
    path = os.path.join(
        os.path.dirname(__file__),
        "prompts",
        filename
    )
    with open(path, "r") as f:
        prompt = f.read()

    # Template placeholders
    prompt = prompt.replace("{journal}", self.journal)
    prompt = prompt.replace("{journal_name}", self.journal_name)

    return prompt

Common Placeholders

{journal} "emj"
{journal_name} "Emergency Medicine Journal"
{ukri_urls} List of guideline URLs
{bmj_urls} List of BMJ policy URLs
Why file-based? Easy to edit, version controlled, supports multi-line formatting

Role Key Permissions Defined In
Terraform Deployment Full IAM, Cognito, S3, Lambda, Bedrock, ECR - (user's AWS role)
AgentCore Runtime bedrock:InvokeModel, s3:*, dynamodb:*, kms:* modules/agentcore_runtime/main.tf
Step Functions lambda:InvokeFunction, logs:*, kms:* infra/terraform/step_functions.tf
Lambda Functions Least-privilege per function modules/lambda/main.tf
Key Point: Each agent gets its own IAM role with only the permissions it needs (least privilege). See modules/agentcore_runtime/ for role definition.

Agent Directory Structure

backend/agents/{agent_name}/
  main.py                 # Simple entry
  entrypoint_streaming.py # Async streaming
  {agent}_agent.py        # Core logic
  models.py               # Pydantic schemas
  utils.py                # Helpers
  requirements.txt        # Dependencies
  Dockerfile              # Container build
  prompts/
    system_prompt.txt     # Main prompt
    decision_prompt.txt   # Sub-prompts
    few_shot_examples.json

Key Components

  • Strands Agent - Framework for agent logic
  • Pydantic Models - Structured outputs
  • FileSessionManager - Conversation history
  • BedrockAgentCoreApp - Runtime wrapper
Developers

Modifying an Agent

1. Edit the Prompt

# prompts/decision_prompt.txt

You are a Scoping Specialist for {journal_name}.

## NEW SECTION: Impact Assessment
Consider potential clinical impact.

## Output
- topic_fits_journal: Yes/No/Uncertain
- potential_impact: High/Medium/Low  # NEW
- decision: Keep/Transfer/Reject
- reasoning: ...

2. Update Output Model

# models.py
class ScopingDecision(BaseModel):
    topic_fits_journal: Literal["Yes", "No", "Uncertain"]
    study_type_fits_journal: Literal["Yes", "No", "Uncertain"]
    potential_impact: Literal["High", "Medium", "Low"]  # NEW
    decision: str
    reasoning: str

3. Redeploy

cd infra/terraform
terraform apply

Creating a New Agent

1. Create Files

mkdir -p backend/agents/my_agent/prompts

# Create:
# - my_agent.py
# - models.py
# - entrypoint_streaming.py
# - utils.py
# - requirements.txt
# - Dockerfile
# - prompts/system_prompt.txt

2. Add Terraform

# agents.tf
module "agentcore_runtime_my_agent" {
  source = "./modules/agentcore_runtime"

  stack_name  = local.stack_name
  agent_name  = "my-agent"
  source_path = "../../backend/agents/my_agent"
  # ...
}

3. Register Agent

# agent_registry.tf
resource "aws_dynamodb_table_item" "my_agent" {
  table_name = module.agent_registry...
  hash_key   = module.agent_registry...

  item = jsonencode({
    agent_name = { S = "my-agent" }
    agent_arn  = { S = module...invoke_arn }
  })
}

Users & Editors

User Workflow

1. Upload PDF
Navigate to /extraction-test, get upload URL, upload file
|
2. Wait for Extraction
System extracts title, abstract, authors, methods
|
3. Start Workflow
Enter S3 key, select model, click Start
|
4. Monitor Progress
Real-time status: pending -> in_progress -> complete
|
5. Review Results
Synthesized decision + editorial commentary
Users & Editors

Interpreting Results

Decision Categories

Proceed to Review Passes all gatekeepers
Transfer Better fit for another journal
Reject Does not meet criteria
Further Investigation Manual review needed

Confidence Scores

80-100: High confidence

60-79: Moderate - may need review

<60: Low - definitely needs review

Red Flags

  • Missing IRB approval
  • Undisclosed conflicts
  • PubPeer concerns
  • Statistical issues
